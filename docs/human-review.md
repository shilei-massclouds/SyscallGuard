# 人工审核

人工审核是第一版的主要质量门禁。自动化可以准备产物，但接受产物必须有明确
的 review 记录。

## 必需字段

每个 sign-off 文件记录：

- `batch_id`
- `step_id`
- `artifact`
- `status`
- `reviewer`
- `reviewed_at`
- `decision_summary`
- `required_changes`
- `follow_up`

## 状态语义

- `pending_human_review`：新产物默认状态。
- `confirmed`：本批次接受该产物。
- `changes_requested`：列出的修改解决前保持阻塞。
- `not_applicable`：按规则跳过，并写明理由。

## 审核清单

- 输入与 batch manifest 和 source index 一致。
- 行为描述足够精确，能够测试或审计。
- 证据路径和 commit 具体明确。
- 不支持或不确定行为已 triage，没有被隐藏。
- coverage matrix 与最终步骤报告一致。

## 屏幕确认格式

每个步骤完成后，助手应在对话中显示一个具体确认块。确认块不写入步骤报告。

确认块顺序：

1. 当前步骤和状态。
2. 需要查看的文件。
3. 本步骤产出或检查的具体列表。
4. 用户需要确认的问题；第 01 步只问“是否同意处理这一批系统调用？”，第 02 步只问“对这批系统调用规格来源的调查结果是否接受？”，第 03 步只问“对这批系统调用检查规则的整理结果是否接受？”，第 04 步只问“是否同意对这些检查规则的分类？”，第 05 步只问“对于这些检查规则的应用目标位置，是否同意？”，第 06 步只问“是否接受这些检查规则的静态审计结果？”，第 07 步只问“是否接受这些未闭合问题的分类与处理决定？”。
5. 可回复动作，例如 `命令：批准进入下一步` 或 `修改 <编号>: ...`；第 01 至 07 步改用无编号的 `修改：<调整内容>`。

示例：

- 第 01 步先列出选中的 syscall 名称，再只问“是否同意处理这一批系统调用？”。
- 第 02 步先列出调查到的规格来源和缺失资料，再只问“对这批系统调用规格来源的调查结果是否接受？”。
- 第 03 步简要列出归一化的 syscall 规则引用、可复用规则和目标映射，再只问“对这批系统调用检查规则的整理结果是否接受？”。
- 第 04 步简要汇总可检查性分类，再只问“是否同意对这些检查规则的分类？”。
- 第 05 步简要汇总映射到的 Starry 目标文件和位置，再只问“对于这些检查规则的应用目标位置，是否同意？”。
- 第 06 步简要汇总静态审计结果，再只问“是否接受这些检查规则的静态审计结果？”。
- 第 07 步简要汇总未闭合问题的分诊结果，再只问“是否接受这些未闭合问题的分类与处理决定？”。
